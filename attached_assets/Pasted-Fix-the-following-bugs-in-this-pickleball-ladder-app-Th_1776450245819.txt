## 6. invalidateQueries() called with no arguments causes a refetch storm

Files: artifacts/pickleball-ladders/src/pages/challenge-detail.tsx, challenge.tsx, availability.tsx, match-detail.tsx, team.tsx, notifications.tsx

Every mutation onSuccess calls qc.invalidateQueries() with no arguments, which refetches every cached query simultaneously.

Replace each one with scoped invalidations. Look at the query keys used by the relevant hooks from @workspace/api-client-react and invalidate only what changed. For example:
- After accepting/declining/cancelling a challenge → invalidate queries with keys matching "challenge" and "ladder"
- After submitting availability → invalidate "availability" and "challenge"  
- After submitting/confirming a score → invalidate "match" and "ladder"
- After accepting a team invitation → invalidate "team", "invitation", and "ladder"
- After marking a notification read → invalidate only "notification"

Look at how the query keys are structured in the generated client (lib/api-client-react/src/) to get the exact key strings to use.

## 7. Position swap in ladder.ts is not wrapped in a transaction

File: artifacts/api-server/src/lib/ladder.ts — swapLadderPositions function

The three-step position swap (set -999, update loser, update winner) is not atomic. If the server crashes between steps the ladder data is corrupted with a team stuck at position -999.

Wrap all three db.update calls inside a db.transaction():
  await db.transaction(async (tx) => {
    await tx.update(ladderStandingsTable).set({ position: -999 }).where(...)
    await tx.update(ladderStandingsTable).set({ position: winnerPos, wins: loserStanding.wins + 1, lastMatchDate: now }).where(...)
    await tx.update(ladderStandingsTable).set({ position: loserPos, losses: winnerStanding.losses + 1, lastMatchDate: now }).where(...)
  });

Also apply the same transaction wrapping to the inactivity drop swap in artifacts/api-server/src/routes/cron.ts (the three db.update calls inside the for loop starting around "Swap positions using temp").

  const activeChallenge = await db.select().from(challengesTable).where(

## 11. Minimum password length not enforced on backend

File: artifacts/api-server/src/routes/auth.ts

The register and change-password routes have no server-side password length check. Only the frontend minLength={8} attribute guards this, which is bypassed by direct API calls.

## 12. Forgot password flow is missing

There is currently no way to recover an account if a player forgets their password.

Backend — add two new routes to artifacts/api-server/src/routes/auth.ts:

POST /auth/forgot-password
- Accept { email } in request body
- Look up the player by email; if not found, still return 200 (don't reveal whether email exists)
- Generate a secure random token: randomBytes(32).toString("hex")
- Store it in a new DB table called password_reset_tokens with columns: token (text, primary key), playerId (uuid), expiresAt (timestamp, set to 1 hour from now)
- Send an email via the existing Resend setup with subject "Reset your Pickleball Ladder password" and a link to [FRONTEND_URL]/reset-password?token=TOKEN
- Return { success: true }

POST /auth/reset-password
- Accept { token, newPassword }
- Look up token in password_reset_tokens; if not found or expiresAt is in the past, return 400 "Invalid or expired reset link"
- Enforce minimum 8 character password
- Hash the new password with the existing hashPassword() function
- Update the player's passwordHash
- Delete the used token from password_reset_tokens
- Return { success: true }

Create the DB migration for the password_reset_tokens table in lib/db/.

Frontend — add two new pages:

artifacts/pickleball-ladders/src/pages/reset-password.tsx
