Build a full-stack web application for a pickleball club ladder system.
Use Next.js (App Router), Supabase (PostgreSQL + Auth + Realtime), and

production-ready, mobile-friendly, and support 200+ members.

=======================================================================
OVERVIEW
=======================================================================

This is a club ladder system for doubles pickleball. Teams of 2 players
challenge other teams to climb a ranked leaderboard over a defined season.
Partners are self-selected and fixed for the entire season. A new partner
can be selected each new season.

- Database + Auth + Realtime: Supabase
- Styling: Tailwind CSS + shadcn/ui components

=======================================================================
DATABASE SCHEMA (Supabase / PostgreSQL)
=======================================================================

-- PLAYERS
create table players (
  id uuid primary key default gen_random_uuid(),
  user_id uuid references auth.users(id) on delete cascade,
  full_name text not null,
  email text not null unique,
  phone text,
  created_at timestamptz default now()
);

-- SEASONS
create table seasons (
  id uuid primary key default gen_random_uuid(),
  name text not null,
  start_date date not null,
  end_date date not null,
  is_active boolean default false,
  created_at timestamptz default now()
);

-- TEAMS (one per season per pair)
create table teams (
  id uuid primary key default gen_random_uuid(),
  season_id uuid references seasons(id) on delete cascade,
  player1_id uuid references players(id) on delete cascade,
  player2_id uuid references players(id) on delete cascade,
  team_name text not null,
  status text default 'pending', -- pending | active | inactive
  created_at timestamptz default now(),
  unique(season_id, player1_id),
  unique(season_id, player2_id)
);

-- TEAM INVITATIONS (player1 invites player2 to form a team)
create table team_invitations (
  id uuid primary key default gen_random_uuid(),
  season_id uuid references seasons(id),
  inviter_id uuid references players(id),
  invitee_id uuid references players(id),
  team_name text not null,
  status text default 'pending', -- pending | accepted | declined
  created_at timestamptz default now()
);

-- LADDER STANDINGS
create table ladder_standings (
  id uuid primary key default gen_random_uuid(),
  season_id uuid references seasons(id) on delete cascade,
  team_id uuid references teams(id) on delete cascade,
  position integer not null,
  wins integer default 0,
  losses integer default 0,
  last_match_date timestamptz,
  created_at timestamptz default now(),
  unique(season_id, position),
  unique(season_id, team_id)
);

-- CHALLENGES
create table challenges (
  id uuid primary key default gen_random_uuid(),
  season_id uuid references seasons(id),
  challenger_team_id uuid references teams(id),
  challenged_team_id uuid references teams(id),
  status text default 'pending',
  -- status flow: pending > accepted > scheduling > scheduled > completed | cancelled | disputed
  created_at timestamptz default now()
);

-- AVAILABILITY (per team, per challenge)
create table availability (
  id uuid primary key default gen_random_uuid(),
  challenge_id uuid references challenges(id) on delete cascade,
  team_id uuid references teams(id),
  slots jsonb not null,
  -- format: [{"date": "2026-04-15", "times": ["18:00", "19:00"]}, ...]
  submitted_at timestamptz default now()
);

-- MATCHES
create table matches (
  id uuid primary key default gen_random_uuid(),
  challenge_id uuid references challenges(id) on delete cascade,
  scheduled_date date not null,
  scheduled_time time not null,
  court_location text not null,
  booked_by_team_id uuid references teams(id),
  status text default 'scheduled', -- scheduled | completed | disputed
  created_at timestamptz default now()
);

-- MATCH SCORES (best of 3 games)
create table match_scores (
  id uuid primary key default gen_random_uuid(),
  match_id uuid references matches(id) on delete cascade,
  game_number integer not null, -- 1, 2, or 3
  team1_score integer not null,
  team2_score integer not null
  -- team1 = challenger, team2 = challenged
);

-- MATCH RESULTS
create table match_results (
  id uuid primary key default gen_random_uuid(),
  match_id uuid references matches(id) on delete cascade,
  winner_team_id uuid references teams(id),
  loser_team_id uuid references teams(id),
  submitted_by_team_id uuid references teams(id),
  confirmed_by_team_id uuid references teams(id),
  submitted_at timestamptz default now(),
  confirmed_at timestamptz,
  auto_confirmed boolean default false,
  dispute_reason text,
  dispute_resolved boolean default false
);

-- INACTIVITY LOG
create table inactivity_drops (
  id uuid primary key default gen_random_uuid(),
  team_id uuid references teams(id),
  season_id uuid references seasons(id),
  old_position integer,
  new_position integer,
  dropped_at timestamptz default now()
);

-- NOTIFICATIONS
create table notifications (
  id uuid primary key default gen_random_uuid(),
  player_id uuid references players(id),
  type text not null,
  message text not null,
  link text,
  is_read boolean default false,
  created_at timestamptz default now()
);

=======================================================================
BUSINESS LOGIC & RULES
=======================================================================

--- SEASONS ---
- Only one season can be active at a time (is_active = true)
- Admin creates seasons with a name, start date, and end date
- At season start, all registered teams are placed at the bottom of the
  ladder in random order
- Teams can register until the season start date

--- TEAM FORMATION ---
- A player searches for their desired partner by name or email
- Player 1 sends a team invitation to Player 2 with a proposed team name
- Player 2 receives an email notification and in-app notification
- Player 2 can accept or decline
- On acceptance, the team is created with status 'active' and placed at
  the bottom of the current season's ladder
- A player can only be on one team per season
- If Player 2 declines, Player 1 is notified and can invite someone else

--- CHALLENGES ---
- A team can only challenge teams that are 2 or 3 spots ABOVE them
  in the current ladder standings
- A team can only have ONE active challenge at a time (either sent or received)
- If a team already has an active challenge, the challenge button is disabled
- Challenge request is sent to the challenged team
- Challenged team receives an email + in-app notification
- Challenged team must accept or decline within 48 hours
  - If no response in 48 hours: challenge auto-expires, challenger is notified
- On acceptance, both teams enter their availability

--- SCHEDULING ---
- After a challenge is accepted, both teams submit their available time slots
- Each team selects multiple date + time combinations from a calendar UI
- The system automatically detects overlapping slots and displays them
- If overlapping slots exist: system shows the common slots and prompts
  one team to confirm a specific slot and enter the court location
- If NO overlapping slots exist: system shows a message:
  "No common availability found. Please contact each other directly to
  arrange a time." and displays both teams' contact info (email + phone)
- Once a court is booked, both teams receive a confirmation email with:
  date, time, location, and both teams' player names and contact info

--- MATCH SCORING ---
- Match format: best of 3 games (first to 2 game wins)
- Standard pickleball scoring (games to 11, win by 2)
- After the match, either team's player can submit the score
- Score entry form: enter score for each game played (2 or 3 games)
- The submitting team also selects the winner
- The opposing team receives an email + in-app notification to confirm
- The opposing team has 48 hours to confirm the score
  - If they confirm: rankings update immediately
  - If they dispute: admin is notified by email, match is flagged, rankings
    do NOT update until admin resolves
  - If no response in 48 hours: score auto-confirms, rankings update,
    both teams are notified

--- RANKING / LADDER LOGIC ---
- Win/loss based. No points system — positions on the ladder are what matter.
- When challenger WINS:
  - Challenger team moves up to the challenged team's old position
  - Challenged team moves down to the challenger's old position
  - (Positions swap)
- When challenger LOSES: no position changes
- After each confirmed result, update ladder_standings accordingly
- Rankings update in real time on the leaderboard

--- INACTIVITY DROPS ---
- A cron job runs every 24 hours
- For each active team in the current season:
  - If last_match_date is NULL, use the season start date as reference
  - If it has been 14 or more days since their last completed match:
    - Drop the team 1 position down (swap with the team below them)
    - Log this in inactivity_drops table
    - Send the team an email warning: "Your team has dropped 1 position
      due to inactivity. Play a match to stay competitive!"
    - Update last_match_date check — only drop once per 14-day window
  - Do not drop a team below the last position on the ladder
- Additionally, teams naturally drop when teams below them win matches
  and swap positions past them

--- DISPUTES ---
- Either team can click "Dispute" when asked to confirm a score
- They must enter a reason for the dispute
- Admin receives an email notification with both teams' contact info,
  the submitted score, and the dispute reason
- Admin can resolve the dispute in the admin panel by:
  - Confirming the submitted score
  - Entering a corrected score
  - Cancelling the match entirely
- Both teams are notified of the resolution by email

=======================================================================
INACTIVITY CRON JOB
=======================================================================

Create an API route: POST /api/cron/inactivity-drop
Protected by a secret header (CRON_SECRET env var)

Logic:
1. Get the active season
2. For each team in the ladder, check last_match_date
   (or season start_date if null)
3. If days since last match >= 14:
   a. Get their current position
   b. Get the team currently at position + 1 (one below them)
   c. If a team exists below them: swap positions
   d. Log the drop in inactivity_drops
   e. Send inactivity email to both players on the team
   f. Update last_match_date check to now to reset the 14-day window
      (use a separate field: last_inactivity_check)
4. Return a summary of all drops made

Set this to run daily via Replit's cron scheduler or an external
cron service hitting the endpoint with the secret header.

=======================================================================
AUTO-CONFIRM CRON JOB
=======================================================================

Create an API route: POST /api/cron/auto-confirm
Protected by a secret header (CRON_SECRET env var)

Logic:
1. Find all match_results where:
   - confirmed_at IS NULL
   - dispute_reason IS NULL
   - submitted_at < now() - interval '48 hours'
2. For each:
   a. Set auto_confirmed = true, confirmed_at = now()
   b. Trigger the ranking update logic (same as manual confirm)
   c. Send auto-confirmed email to both teams

Set this to run every hour.

=======================================================================
CHALLENGE EXPIRY CRON JOB
=======================================================================

Create an API route: POST /api/cron/expire-challenges
Protected by a secret header (CRON_SECRET env var)

Logic:
1. Find all challenges where:
   - status = 'pending'
   - created_at < now() - interval '48 hours'
2. For each:
   a. Set status = 'cancelled'
   b. Send challenge-expired email to challenger team

Set this to run every hour.

- After registration, user must verify email before accessing the app

=======================================================================
IMPORTANT IMPLEMENTATION NOTES
=======================================================================

1. All database mutations that affect ladder positions should be wrapped
   in transactions to prevent race conditions.

2. When positions swap (after a match), lock both rows before updating
   to prevent concurrent swaps from corrupting the ladder.

3. The leaderboard page should be server-rendered for SEO and initial
   load speed, then hydrated with Supabase Realtime for live updates.

4. Use Next.js Server Actions or API routes for all mutations — do not
   expose Supabase service role key to the client.

5. Validate all challenge eligibility server-side (do not trust
   client-side checks alone) — check positions in the DB at the moment
   of challenge submission.

6. For the availability overlap detection: normalize all time slots to
   UTC before storing and comparing.

7. Ensure the admin panel is completely inaccessible to non-admin users
   at the middleware level (not just UI-level hiding).

8. All emails should be sent asynchronously (do not block the API
   response waiting for the email to send).

9. Add a /api/health endpoint that returns {status: "ok"} for
   uptime monitoring.

10. Include a seed script that creates: one test season, 10 test teams
    with fake players, and populates the ladder — so the app is
    immediately demonstrable after setup.
